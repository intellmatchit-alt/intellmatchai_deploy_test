1. Executive Overview

IntellMatch (formerly P2P Network) is a Progressive Web Application (PWA) for professional networking and relationship intelligence. It enables users to manage contacts, scan business cards via OCR, discover meaningful connections through AI-powered matching, manage projects, pitches, deals, job opportunities, and collaborate with their network.

Core User Flows

  • Contact Acquisition: Scan business cards (OCR) → Review/confirm fields → Enrich via PDL → Auto-match against network
  • Intelligent Matching: 10-signal scoring engine (semantic similarity, goals, skills, sectors, network) → Ranked recommendations → Explainable AI reasons
  • Project Collaboration: Create project → Define needs/skills → AI matches contacts → Send collaboration requests → Track via ledger
  • Pitch-to-Network: Upload pitch deck → AI extracts key info → Match against contacts → Generate outreach messages
  • Job Matching: Define hiring needs or availability → AI ranks candidates → Itemized scoring per criterion
  • Deal Matching: Create buy/sell request → AI matches counterparties → Facilitate introductions
  • Event Networking: Create event → QR-based registration → Attendee matching → Suggested introductions

AI Matching Overview

The matching engine uses a 10-signal weighted scoring algorithm (v7). The primary signal is semantic similarity (30%) computed via OpenAI embeddings, followed by goal alignment (18%) and skill match (18%). Network proximity is computed via Neo4j graph queries. Optional Recombee integration provides hybrid ML recommendations. All matches include LLM-generated explanations (Groq/Gemini/OpenAI with auto-fallback) and suggested ice-breaker messages.

4.5 API Service Layer

All API calls go through a centralized client at lib/api/client.ts with automatic token refresh, error handling, and organization context headers.

API Client Architecture

// Token management
getAccessToken()    // Read from localStorage: p2p_access_token
getRefreshToken()   // Read from localStorage: p2p_refresh_token
setTokens(access, refresh)  // Store to localStorage
clearTokens()       // Remove from localStorage

// Request flow
apiRequest(endpoint, options) {
  1. Add Authorization: Bearer {accessToken} header
  2. Add X-Organization-Id header (if active org)
  3. Send request
  4. On 401: attempt token refresh (queue concurrent requests)
  5. On success: return parsed JSON
  6. On error: throw ApiError with status, code, message
}

5. Backend Documentation

5.1 Clean Architecture

The backend follows Clean Architecture with four layers. Dependencies point inward: Presentation → Application → Domain ← Infrastructure.

Layer Responsibilities

  • Domain Layer (src/domain/): Entities, repository interfaces, service interfaces, value objects, domain events. No external dependencies.
  • Application Layer (src/application/): Use cases (business operations), DTOs, application interfaces. Orchestrates domain objects.
  • Infrastructure Layer (src/infrastructure/): Database implementations (Prisma, Neo4j, Redis), external service integrations, queue workers, WebSocket, cache.
  • Presentation Layer (src/presentation/): HTTP controllers, routes, middleware, validators, request/response schemas.

Bootstrap Sequence (src/main.ts)

1. Load environment config (Zod validated)
2. Initialize Prisma (MySQL)
3. Initialize Redis
4. Initialize AI Services Container (ServiceContainer)
5. Apply security middleware (Helmet, CORS, compression)
6. Apply rate limiters (general: 100/min, auth: 5/15min, scan: 10/min)
7. Mount API routes at /api/v1
8. Mount error handler middleware
9. Initialize BullMQ workers (18 workers)
10. Initialize Socket.IO WebSocket server
11. Start HTTP server (default port 3001)
12. Register SIGTERM/SIGINT graceful shutdown handlers

6.3 Neo4j (Graph Database)

Used for network proximity scoring in the matching engine. Stores user-contact relationships as a graph to compute 2nd and 3rd degree connections.

  • Nodes: User, Contact
  • Edges: KNOWS, CONNECTED_TO, INTRODUCED_BY
  • Queries: Shortest path between users, 2nd/3rd degree connections, cluster detection
  • Scoring: 2nd degree = 80 points, 3rd degree = 50 points (mapped to network proximity weight of 6%)
  • Visualization: D3.js force-directed graph in frontend /map page

7.2 Login Flow

1. User submits email + password
2. Backend checks Redis for lockout (5 failed attempts = 15 min lockout)
3. Validates credentials against bcrypt hash (12 rounds)
4. Generates access + refresh token pair
5. Stores refresh token in database (RefreshToken model)
6. Updates user.lastLoginAt
7. Returns: { user, accessToken, refreshToken, expiresAt }
8. Frontend stores tokens in localStorage
9. Redirects to /dashboard

7.3 Token Refresh Flow

1. API client gets 401 response
2. Sets isRefreshing flag (prevents concurrent refresh)
3. Queues pending requests as subscribers
4. Calls POST /auth/refresh with refreshToken
5. Backend validates refresh token against database
6. Issues new access + refresh token pair
7. Stores new refresh token, revokes old one
8. Frontend updates localStorage
9. Retries all queued requests with new access token
10. If refresh fails: clears tokens, redirects to /login

8. AI Matching System

The matching system is the core intellectual property of IntellMatch. It uses a multi-signal weighted scoring algorithm to rank professional connections by relevance and potential value.

8.1 Scoring Formula (v7)

File: backend/src/infrastructure/external/matching/DeterministicMatchingService.ts

SignalWeightData SourceScoring Logic
Semantic Similarity30%OpenAI embeddings of profile textCosine similarity of bio + skills + interests + goals embeddings, scaled 0-100
Goal Alignment18%User goals + contact profile patternsPattern-matches goals (MENTORSHIP, INVESTMENT, PARTNERSHIP, HIRING, etc.) against job titles and industries
Skill Match18%UserSkills, ContactSkillsCount overlapping skills, weighted by proficiency. Formula: count / max * 100
Sector Overlap10%UserSectors, ContactSectorsCount shared sectors. Formula: count / max * 100
Complementary Skills8%Complementary skills matrix (30+ pairs)100 if complementary pair found, 0 otherwise. E.g., Frontend ↔ Backend
Network Proximity6%Neo4j graph queries2nd degree: 80 points, 3rd degree: 50 points, no connection: 0
Recency3%Last interaction timestamp0-30 days: 100, 30-60 days: linear decay, >60 days: 0
Interaction Frequency3%Interaction count (last 30 days)Formula: min(count, 10) / 10 * 100
Interest Overlap2%UserInterests, ContactInterestsCount shared interests. Formula: count / max * 100
Hobby Overlap2%UserHobbies, ContactHobbiesCount shared hobbies. Formula: count / max * 100

Final Score = Sum of (signal_score * weight) for all 10 signals, rounded to 0-100.

Weaknesses & Improvement Suggestions

  • Embedding dependency: Semantic similarity (30% weight) requires OpenAI API. If unavailable, largest signal is lost.
  • Cold start problem: New contacts with no skills/sectors/goals get low scores across most signals.
  • Neo4j dependency: Network proximity (6%) requires Neo4j to be synced. No automatic sync mechanism documented.
  • Single match recalculation: Changing one contact's skills doesn't automatically recalculate all affected matches.
  • No feedback loop: User actions (accept/reject match) don't directly influence future scoring weights.
  • Suggestion: Add user feedback signals to scoring, implement scheduled batch re-scoring, add embedding cache layer.

14. Important Business Logic

12.1 Contact Import Pipeline

Multi-stage BullMQ pipeline for batch imports:

Stage 1: NORMALIZE  → Parse CSV/vCard, extract fields, normalize formats
Stage 2: DEDUPE     → Generate identity keys, find duplicates, merge
Stage 3: ENRICH     → PDL enrichment (if enabled)
Stage 4: TAG        → Auto-tag sectors, skills, interests
Stage 5: SUMMARY    → Generate AI bio summary
Stage 6: MATCH      → Compute match scores against user profile

12.3 Wallet & Points Economy

  • Points: In-app currency for premium actions (scanning, enrichment, collaboration)
  • Point Packs: Purchasable via PayTabs (various tiers)
  • Deduction triggers: OCR scan, PDL enrichment, collaboration requests
  • Balance tracking: Wallet model with WalletTransaction audit trail
  • Decimal support: Migration added decimal precision for accurate accounting

12.4 Collaboration Workflow

1. User A finds a match (contact, project, opportunity)
2. User A sends CollaborationRequest to User B
3. User B receives notification (in-app + email)
4. User B accepts/declines request
5. On accept: CollaborationLedger created (tracks points)
6. CollaborationSession initiated for ongoing work
7. Warm introductions facilitated (WarmIntroRequest)
8. Points earned/spent tracked in CollaborationTransaction

12.5 SDAIA PDPL Compliance

Saudi Arabia's Personal Data Protection Law compliance features:

  • ConsentLog: Tracks user consent for data processing
  • SdaiaProcessingRecord: Audits all personal data processing
  • Retention policies: Configurable via SDAIA_RETENTION_DAYS
  • DPO contact: Configured via SDAIA_DPO_EMAIL
  • Controller info: SDAIA_CONTROLLER_NAME for legal compliance

12.6 Onboarding Flow

1. User registers → email verification sent
2. First login → onboarding wizard displayed
3. Step 1: Complete profile (name, company, bio)
4. Step 2: Select sectors (primary + secondary)
5. Step 3: Add skills with proficiency levels
6. Step 4: Define goals (what they're looking for)
7. Step 5: Add interests and hobbies
8. On completion: onboardingCompletedAt set, matching begins
